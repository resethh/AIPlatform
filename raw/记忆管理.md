## 一、OpenClaw 记忆模型拆解

### 1.1 OpenClaw 的四层记忆体系

从源码看，OpenClaw 的记忆是一个 **纯文件系统 + Agent 自管理** 的模型：

```
┌─────────────────────────────────────────────────────────────────┐
│                    OpenClaw 记忆架构                              │
│                                                                 │
│  ┌─────────────── 短期记忆 ───────────────┐                      │
│  │                                        │                      │
│  │  Session Transcript (JSONL)            │                      │
│  │  ┌──────────────────────────┐          │                      │
│  │  │ {type:"message", ...}    │          │                      │
│  │  │ {type:"message", ...}    │ ← 逐行   │                      │
│  │  │ {type:"message", ...}    │   追加    │                      │
│  │  └──────────────────────────┘          │                      │
│  │  位置: state/agents/<id>/sessions/     │                      │
│  │  生命周期: Session 存续期               │                      │
│  │  重置: /new 或 /reset 命令              │                      │
│  └────────────────────────────────────────┘                      │
│                                                                 │
│  ┌─────────────── 中期记忆 ───────────────┐                      │
```

### 1.2 关键实现细节（源码）

**Session Transcript:**
- 格式: JSONL (每行一个 JSON 对象)
- 文件: `{sessionId}.jsonl`，支持 `.reset.*` 后缀的归档
- 写锁: `SessionWriteLock`，基于文件锁，支持重入，防止并发写入
- 重置策略: `daily`（每天重置）/ `idle`（空闲 N 分钟后重置）/ `manual`（手动 /new）
- 支持按渠道、按聊天类型（group/direct/thread）配置不同重置策略

**Session Memory Hook:**
- 触发: `/new` 或 `/reset` 命令
- 流程: 读取最近 15 条消息 → LLM 生成 1-2 词 slug → 保存为 `memory/YYYY-MM-DD-{slug}.md`
- Slug 生成: 调用 `generateSlugViaLLM()`，用最便宜的模型

**Workspace Context Injection:**
- 在 system prompt 中注入 `AGENTS.md`、`SOUL.md`、`USER.md` 等
- `MEMORY.md` 仅在 main session（直接聊天）中注入，群聊不注入（安全隔离）
- Agent 通过 `read`/`write`/`edit` 工具自行维护这些文件

### 1.3 OpenClaw 模型的优势与局限

| 维度 | 优势 | 局限 |
|---|---|---|
| 简单性 | 纯文件系统，零依赖 | 不支持语义搜索 |
| 隐私 | 全部本地，数据不出机器 | 不支持跨设备同步 |
| 灵活性 | Agent 自由读写，无限扩展 | 无结构化，靠 Agent 自律 |
| 成本 | 零额外成本 | 全量加载到 prompt，token 浪费 |
| 多用户 | 单用户完美 | **完全不支持多用户/多租户** |
| 搜索 | 靠 Agent 读文件 | **无向量检索，大量记忆时效率极低** |
| 一致性 | 文件锁保证 | **无事务，并发场景可能丢数据** |

## 二、AI 中台记忆体系需要什么

### 2.1 核心需求差异

```
OpenClaw: 一个人的私人助手记忆
AI 中台:  一个组织的多 Agent 多用户记忆系统

需要解决:
1. 多用户隔离 — A 用户的记忆不能泄露给 B
2. 多 Agent 共享 — 同一用户和多个 Agent 交互，记忆需要部分共享
3. 组织级知识 — 某些"记忆"是组织共有的（产品手册、流程规范等）
4. 语义检索 — 几千条记忆不能全塞 prompt，需要按相关性检索
5. 记忆生命周期 — 自动淘汰过期记忆，自动提炼长期记忆
6. 合规要求 — 记忆数据分类、保留期限、删除权（GDPR/数据安全法）
```

### 2.2 记忆分类体系

```
┌─────────────────────────────────────────────────────────────────────┐
│                     AI 中台记忆分类                                   │
│                                                                     │
│  按 所有权:                                                          │
│  ├── 用户私有记忆 (User Memory)    ← 个人偏好、历史对话、习惯        │
│  ├── Agent 私有记忆 (Agent Memory)  ← Agent 的工作经验、学到的技巧   │
│  ├── 共享记忆 (Shared Memory)      ← 用户↔Agent 的互动积累          │
│  └── 组织记忆 (Org Memory)         ← 组织知识库、流程规范、最佳实践  │
│                                                                     │
│  按 时效:                                                            │
│  ├── 工作记忆 (Working Memory)     ← 当前对话上下文 (~分钟级)        │
│  ├── 短期记忆 (Short-term)         ← 近期交互摘要 (~天级)            │
│  ├── 长期记忆 (Long-term)          ← 提炼后的持久记忆 (~月/年级)     │
│  └── 永久记忆 (Permanent)          ← 组织知识、不过期 (~永久)        │
│                                                                     │
│  按 结构:                                                            │
│  ├── 事实记忆 (Factual)            ← "用户喜欢简洁风格"              │
│  ├── 事件记忆 (Episodic)           ← "2026-04-10 处理了CRM迁移"     │
│  ├── 程序记忆 (Procedural)         ← "查CRM要先加region参数"        │
│  └── 语义记忆 (Semantic)           ← 知识图谱、概念关系              │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.3 对比 OpenClaw 的改造要点

| OpenClaw 组件 | OpenClaw 实现 | AI 中台改造 |
|---|---|---|
| **Session Transcript** (JSONL) | 本地文件，逐行追加 | **Redis (工作记忆) + PG (持久化)**；Redis 存当前上下文，PG 做归档 |
| **Session Memory Hook** (/new 时保存) | 最近 15 条消息 → markdown 文件 | **自动摘要 + 事实提取 → memories 表**；不是原始消息而是提炼后的记忆条目 |
| **MEMORY.md** (长期精炼记忆) | Agent 手动维护的 markdown | **记忆提炼器 (Distiller) 自动维护**；定期把短期记忆提炼为长期记忆，存入 memories 表 |
| **USER.md** (用户画像) | Agent 手动维护 | **用户画像表 (user_profiles)**；结构化存储，跨 Agent 共享，Agent 可读但按权限写 |
| **Workspace Files** (SOUL/TOOLS 等) | 每次注入 system prompt | **Agent 配置 + Skill 元数据**；不再靠文件，走数据库和 API |
| **文件锁** (SessionWriteLock) | 文件级锁 | **Redis 分布式锁 + PG 事务**；支持多 Worker 并发 |
| **全量加载** (prompt 注入) | 每次请求全量注入 workspace 文件 | **按需检索**；只注入与当前对话相关的记忆片段 |
| **LLM Slug Generator** | 用 LLM 生成文件名 | **沿用思路**，但用于生成记忆摘要标题而非文件名 |

## 四、核心模块设计

### 4.1 记忆写入器 (Memory Writer)

```python
# memory/writer.py

class MemoryWriter:
    """
    记忆写入 — 从对话和工具调用中提取记忆
    
    OpenClaw 的做法: /new 时把最近 15 条消息原样保存为 markdown
    AI 中台改造: 实时提取 + 异步提炼
    """
    
    # ── 实时提取（同步，每轮对话后执行）──
    
    async def extract_from_turn(self, turn: ConversationTurn,
                                 context: MemoryContext) -> List[MemoryEntry]:
        """
        每轮对话后，从用户消息和 Agent 响应中提取记忆
        
        提取类型:
        1. 用户明确声明 → "记住我喜欢简洁的回复" → factual, importance=0.9
        2. 用户偏好推断 → 用户总是问CRM → factual, importance=0.5
        3. 重要事实 → "我们的CRM今天迁移到了新地址" → factual, importance=0.8
        4. 事件记录 → 用户发起了一次审批 → episodic, importance=0.3
        5. 工具使用经验 → crm_query 需要 region 参数才不报错 → procedural, importance=0.7
        """
        entries = []
        
        # 1. 规则提取（快速，零成本）
        rule_entries = self._rule_based_extract(turn)
        entries.extend(rule_entries)
        
        # 2. LLM 提取（异步，只对重要对话）
        if self._should_llm_extract(turn):
            llm_entries = await self._llm_extract(turn, context)
            entries.extend(llm_entries)
        
        # 3. 去重（与已有记忆对比）
        deduped = await self._deduplicate(entries, context)
        
        # 4. 写入
        for entry in deduped:
            entry.tenant_id = context.tenant_id
            entry.embedding = await self.embedder.embed(entry.content)
            await self.store.insert(entry)
        
        return deduped
    
    
    def _rule_based_extract(self, turn: ConversationTurn) -> List[MemoryEntry]:
        """规则提取（零成本）"""
        entries = []
        text = turn.user_message.text
        
        # 明确请求记忆
        remember_patterns = [
            r'(?:记住|记下|记一下|remember|note)\s*[:：]?\s*(.+)',
            r'(?:以后|下次|今后)\s*(.+)',
            r'我(?:喜欢|偏好|习惯|总是|不要|别)\s*(.+)',
        ]
        
        for pattern in remember_patterns:
            match = re.search(pattern, text)
            if match:
                entries.append(MemoryEntry(
                    content=match.group(1).strip(),
                    content_type="factual",
                    source_type="user_explicit",
                    importance=0.9,
                    memory_tier="long_term",
                ))
        
        return entries
    
    
    async def _llm_extract(self, turn: ConversationTurn,
                            context: MemoryContext) -> List[MemoryEntry]:
        """LLM 提取（有成本，选择性调用）"""
        
        prompt = f"""从以下对话中提取值得记忆的信息。

对话:
用户: {turn.user_message.text[:500]}
助手: {turn.agent_response.text[:500]}

提取规则:
1. 只提取有长期价值的信息（用户偏好、重要事实、工作流经验）
2. 不提取一次性的查询内容
3. 每条记忆一句话，简洁明了
4. 标注类型: factual(事实) / episodic(事件) / procedural(经验)
5. 标注重要度: 0.0-1.0

如果没有值得记忆的信息，返回空列表。

返回 JSON: [{{"content": "...", "type": "...", "importance": 0.0-1.0}}]"""
        
        response = await self.llm.chat(
            model="qwen-turbo",  # 最便宜
            messages=[{"role": "user", "content": prompt}],
            temperature=0,
            max_tokens=300,
        )
        
        items = json.loads(response.content)
        return [
            MemoryEntry(
                content=item["content"],
                content_type=item["type"],
                source_type="agent_inferred",
                importance=item["importance"],
                memory_tier="short_term",
            )
            for item in items
            if item.get("content")
        ]
    
    
    async def _deduplicate(self, new_entries: List[MemoryEntry],
                            context: MemoryContext) -> List[MemoryEntry]:
        """
        与已有记忆去重
        
        策略:
        1. 向量相似度 > 0.95 → 认为重复，跳过
        2. 向量相似度 0.85-0.95 → 可能是更新，保留新的并标记旧的为 superseded
        3. 向量相似度 < 0.85 → 新记忆，直接插入
        """
        deduped = []
        
        for entry in new_entries:
            embedding = await self.embedder.embed(entry.content)
            
            similar = await self.store.search_similar(
                embedding=embedding,
                tenant_id=context.tenant_id,
                owner_id=context.owner_id,
                threshold=0.85,
                limit=3,
            )
            
            if similar and similar[0].score > 0.95:
                # 重复，跳过
                continue
            elif similar and similar[0].score > 0.85:
                # 更新，标记旧的为 superseded
                await self.store.mark_superseded(similar[0].memory_id, entry)
                deduped.append(entry)
            else:
                deduped.append(entry)
        
        return deduped
```

### 4.2 记忆检索器 (Memory Retriever)

```python
# memory/retriever.py

class MemoryRetriever:
    """
    记忆检索 — 根据当前对话上下文检索相关记忆
    
    OpenClaw 的做法: 每次全量注入 MEMORY.md + 最近的 memory/ 文件
    AI 中台改造: 混合检索（向量 + 关键词 + 时间衰减 + 重要度加权）
    """
    
    async def retrieve(self, query: str, context: MemoryContext,
                        max_results: int = 10) -> List[RetrievedMemory]:
        """
        主检索入口 — 混合检索策略
        
        三路召回 → 融合排序 → 返回
        """
        
        # ── 路径 1: 向量语义检索（捕获语义相关）──
        query_embedding = await self.embedder.embed(query)
        semantic_results = await self.store.vector_search(
            embedding=query_embedding,
            filters=self._build_access_filters(context),
            limit=max_results * 2,  # 多召回，后面融合
        )
        
        # ── 路径 2: 关键词检索（捕获精确匹配）──
        keywords = self._extract_keywords(query)
        keyword_results = await self.store.keyword_search(
            keywords=keywords,
            filters=self._build_access_filters(context),
            limit=max_results * 2,
        )
        
        # ── 路径 3: 时间+重要度检索（捕获近期重要记忆）──
        recent_important = await self.store.recent_important(
            filters=self._build_access_filters(context),
            hours=48,           # 近 48 小时
            min_importance=0.7, # 高重要度
            limit=max_results,
        )
        
        # ── 融合排序 ──
        all_results = self._merge_and_rank(
            semantic_results, keyword_results, recent_important,
            context=context,
        )
        
        # ── 去重 + 截断 ──
        final = self._deduplicate_and_limit(all_results, max_results)
        
        # ── 记录访问（用于重要度衰减/增强）──
        await self._log_access(final, context)
        
        return final
    
    
    def _build_access_filters(self, context: MemoryContext) -> dict:
        """
        构建访问权限过滤器
        
        可见记忆范围:
        1. 用户自己的私有记忆 (owner_type=user, owner_id=user_id)
        2. 当前 Agent 的记忆 (owner_type=agent, owner_id=agent_id)
        3. 用户↔Agent 的共享记忆 (owner_type=shared)
        4. 用户所在组织节点的组织记忆 (owner_type=org, 组织树继承)
        5. 公开记忆 (data_classification=public)
        """
        return {
            "tenant_id": context.tenant_id,
            "status": "active",
            "$or": [
                # 用户私有
                {"owner_type": "user", "owner_id": context.user_id},
                # Agent 记忆
                {"owner_type": "agent", "owner_id": context.agent_id},
                # 共享记忆
                {"owner_type": "shared", "owner_id": f"shared:{context.user_id}:{context.agent_id}"},
                # 组织记忆（继承链上的所有节点）
                {"owner_type": "org", "owner_id": {"$in": context.org_path_node_ids}},
                # 公开
                {"data_classification": "public"},
            ],
        }
    
    
    def _merge_and_rank(self, semantic, keyword, recent_important, context):
        """
        融合排序 — Reciprocal Rank Fusion (RRF)
        
        score = Σ (1 / (k + rank_i)) × weight_i
        
        权重:
        - 语义相关度: 0.4
        - 关键词匹配: 0.2
        - 时间近度: 0.15
        - 重要度: 0.15
        - 访问频率: 0.1
        """
        k = 60  # RRF 常数
        scores = {}
        
        # 语义排名
        for rank, item in enumerate(semantic):
            scores.setdefault(item.memory_id, 0)
            scores[item.memory_id] += 0.4 * (1.0 / (k + rank))
        
        # 关键词排名
        for rank, item in enumerate(keyword):
            scores.setdefault(item.memory_id, 0)
            scores[item.memory_id] += 0.2 * (1.0 / (k + rank))
        
        # 近期重要
        for rank, item in enumerate(recent_important):
            scores.setdefault(item.memory_id, 0)
            scores[item.memory_id] += 0.15 * (1.0 / (k + rank))
        
        # 额外加权：重要度 + 时间衰减
        all_items = {item.memory_id: item for items in [semantic, keyword, recent_important] for item in items}
        for mid, score in scores.items():
            item = all_items.get(mid)
            if item:
                # 重要度加成
                scores[mid] += 0.15 * item.importance
                # 时间衰减（越新越好，半衰期 7 天）
                age_days = (datetime.now() - item.created_at).days
                time_factor = 0.5 ** (age_days / 7)
                scores[mid] += 0.1 * time_factor
        
        # 按分数降序排列
        ranked = sorted(scores.items(), key=lambda x: x[1], reverse=True)
        return [RetrievedMemory(memory_id=mid, score=score, item=all_items[mid]) for mid, score in ranked if mid in all_items]
```

### 4.3 记忆提炼器 (Memory Distiller)

```python
# memory/distiller.py

class MemoryDistiller:
    """
    记忆提炼 — 周期性将短期记忆提炼为长期记忆
    
    OpenClaw 的做法: 靠 Agent 在心跳时手动审查 memory/ 文件并更新 MEMORY.md
    AI 中台改造: 自动化提炼管道
    
    类比人类记忆:
    - 睡眠时大脑整理白天的记忆 → 我们用定时任务
    - 重要的记忆强化 → 高访问频率/高重要度的记忆提升
    - 不重要的记忆淡忘 → 低访问/低重要度的记忆降级或归档
    """
    
    async def run_distillation(self, tenant_id: str, owner_id: str):
        """
        定时运行（建议每天凌晨 / 每 6 小时）
        
        流程:
        1. 收集该用户的短期记忆
        2. 聚类相关记忆
        3. 合并/提炼为长期记忆
        4. 标记原始短期记忆为 superseded
        5. 衰减不常用的长期记忆
        """
        
        # ── Step 1: 收集短期记忆 ──
        short_term = await self.store.list_memories(
            tenant_id=tenant_id,
            owner_id=owner_id,
            memory_tier="short_term",
            status="active",
            min_age_hours=24,       # 至少 24 小时前的（太新的不动）
            limit=100,
        )
        
        if len(short_term) < 3:
            return  # 太少不值得提炼
        
        # ── Step 2: 聚类 ──
        clusters = self._cluster_by_topic(short_term)
        
        # ── Step 3: 逐簇提炼 ──
        for cluster in clusters:
            if len(cluster) < 2:
                # 单条记忆直接提升为长期
                await self.store.update_tier(cluster[0].memory_id, "long_term")
                continue
            
            # 用 LLM 合并提炼
            distilled = await self._distill_cluster(cluster)
            
            if distilled:
                # 插入提炼后的长期记忆
                distilled.memory_tier = "long_term"
                distilled.source_type = "distillation"
                distilled.importance = max(m.importance for m in cluster) * 1.1  # 略提升
                distilled.embedding = await self.embedder.embed(distilled.content)
                
                new_id = await self.store.insert(distilled)
                
                # 标记原始记忆为已取代
                for mem in cluster:
                    await self.store.mark_superseded(mem.memory_id, new_id)
        
        # ── Step 4: 衰减不常用记忆 ──
        await self._decay_unused_memories(tenant_id, owner_id)
    
    
    async def _distill_cluster(self, cluster: List[MemoryEntry]) -> Optional[MemoryEntry]:
        """用 LLM 将一组相关记忆提炼为一条"""
        
        memories_text = "\n".join([
            f"- [{m.content_type}] {m.content} (重要度:{m.importance:.1f}, 来源:{m.source_type})"
            for m in cluster
        ])
        
        prompt = f"""将以下相关记忆提炼为一条简洁的长期记忆:

{memories_text}

规则:
1. 合并重复信息
2. 保留最重要的细节
3. 如果有矛盾，以最新的为准
4. 一段话，不超过 200 字
5. 标注类型: factual / episodic / procedural

返回 JSON: {{"content": "...", "type": "...", "tags": [...]}}"""
        
        response = await self.llm.chat(
            model="qwen-turbo",
            messages=[{"role": "user", "content": prompt}],
            temperature=0,
            max_tokens=300,
        )
        
        data = json.loads(response.content)
        return MemoryEntry(
            content=data["content"],
            content_type=data["type"],
            tags=data.get("tags", []),
        )
    
    
    async def _decay_unused_memories(self, tenant_id: str, owner_id: str):
        """
        记忆衰减 — 长时间未被检索的记忆逐渐降低重要度
        
        半衰期模型:
        - 短期记忆: 未检索超 7 天 → 归档
        - 长期记忆: 未检索超 90 天 → 重要度减半
        - 长期记忆: 重要度 < 0.1 → 归档
        - 永久记忆: 不衰减
        """
        
        # 短期记忆过期清理
        await self.store.archive_memories(
            tenant_id=tenant_id,
            owner_id=owner_id,
            memory_tier="short_term",
            not_accessed_days=7,
        )
        
        # 长期记忆衰减
        stale_long_term = await self.store.list_memories(
            tenant_id=tenant_id,
            owner_id=owner_id,
            memory_tier="long_term",
            status="active",
            not_accessed_days=90,
        )
        
        for mem in stale_long_term:
            new_importance = mem.importance * 0.5
            if new_importance < 0.1:
                await self.store.archive(mem.memory_id)
            else:
                await self.store.update_importance(mem.memory_id, new_importance)
```

```python
# worker/agent_worker.py 中的修改

class AgentWorker:
    
    async def run_agent_loop_with_memory(self, agent, messages, task, trace_id,
                                          resolved_skills):
        """改进后的 Agent Loop — 集成记忆系统"""
        
        context = MemoryContext(
            tenant_id=task["tenant_id"],
            user_id=task["user_id"],
            agent_id=task["agent_id"],

        relevant_memories = await self.memory_retriever.retrieve(
            query=user_query,
            context=context,
            max_results=10,
        )
        
        if relevant_memories:
            memory_block = self._format_memories_for_prompt(relevant_memories)
            # 注入到系统消息中（类似 OpenClaw 注入 MEMORY.md）
            messages[0]["content"] += f"\n\n{memory_block}"
        
        # ── 2. 正常执行 Agent Loop ──
        result = await self._run_loop(agent, messages, task, trace_id, resolved_skills)
        
        # ── 3. 从本轮对话提取新记忆 ──
        turn = ConversationTurn(
            user_message=messages[-1],
            agent_response=result.final_message,
            tool_calls=result.tool_calls,
        )
        
        new_memories = await self.memory_writer.extract_from_turn(turn, context)
        
        return result
    
    
    def _format_memories_for_prompt(self, memories: List[RetrievedMemory]) -> str:
        """
        格式化记忆为 prompt 片段
        
        OpenClaw 的做法: 直接注入 MEMORY.md 全文
        AI 中台: 只注入检索到的相关条目，带元数据
        """
        lines = [
            "## Relevant Memories",
            "(以下是与当前对话相关的历史记忆，供你参考)",
            ""
        ]
        
        for i, mem in enumerate(memories, 1):
            tier_emoji = {
                "long_term": "🧠",
                "short_term": "📝",
                "permanent": "📚",
            }.get(mem.item.memory_tier, "💭")
            
            lines.append(
                f"{tier_emoji} [{mem.item.content_type}] {mem.item.content}"
                f" (相关度:{mem.score:.0%})"
            )
        
        return "\n".join(lines)
```

### 5.2 需要改造的

| OpenClaw 设计 | 问题 | 改造方案 |
|---|---|---|
| **文件系统存储** | 不支持搜索、不支持并发、不支持多用户 | PostgreSQL + pgvector + Redis |
| **全量注入 prompt** | 浪费 token，记忆多了就超限 | 向量检索按需注入 |

| **无分类** | 所有记忆平等 | 四维分类（所有权×时效×结构×安全等级） |
| **单用户** | 只有一个人的记忆 | 多租户×多用户×多 Agent 隔离 |
| **无审计** | 不知道谁访问了什么记忆 | 记忆访问日志 |

### 5.3 需要新增的

### 7.2 Hermes Agent 记忆系统详解

**项目**: NousResearch/hermes-agent — "The self-improving AI agent"

#### 核心架构：双文件 + 硬限制 + Session FTS5 + 8 个 Provider 插件

| 组件 | 实现 | 关键细节 |
|---|---|---|
| **MEMORY.md** | Agent 笔记 | 硬限制 2,200 字符 (~800 tokens)，满了必须合并/替换 |
| **USER.md** | 用户画像 | 硬限制 1,375 字符 (~500 tokens)，Agent 自动维护 |
| **memory tool** | add/replace/remove | 子串匹配定位（给唯一片段即可），无 read action |
| **Frozen Snapshot** | Session 开始时快照注入 | Session 中间的修改写盘但不刷 prompt（保护 LLM prefix cache） |
| **Session Search** | SQLite + FTS5 | 所有历史会话全文索引，搜索时用 Gemini Flash 做摘要 |
| **安全扫描** | 注入检测 | 写入前检查 prompt injection、credential exfiltration、不可见 Unicode |
| **去重** | 精确重复检测 | 完全相同的内容自动拒绝 |
| **外部 Provider** | 8 个插件 | Honcho / OpenViking / Mem0 / Hindsight / Holographic / RetainDB / ByteRover / Supermemory |

#### Hermes 关键设计决策

1. **有界记忆（Bounded Memory）** — 给记忆设硬字符上限，满了就得合并旧条目。这强制了记忆质量，让 LLM 自己做"什么值得保留"的价值判断。
2. **冻结快照模式** — System prompt 只在 session start 时加载记忆，中间不变。保护 LLM KV cache，性能好；代价是当前 session 看不到刚保存的记忆。
3. **Session Search 是独立层** — MEMORY.md 是"总是可见的关键事实"(~1300 tokens)，Session Search 是"按需回忆过去的对话"(无限量)。分层很聪明。
4. **外部记忆 Provider** — 内置记忆是基础，可挂载 Honcho（辩证式用户建模）、Mem0（自动事实提取）等作为增强层。
5. **Proactive Memory** — Agent 自动保存偏好、纠正、经验，不需要用户说"记住"。

#### Hermes 外部 Provider 关键特性

| Provider | 核心能力 | 衰减/更新机制 |
|---|---|---|
| **Honcho** | 辩证推理用户建模，跨 session 上下文注入 | dialecticCadence 控制 LLM reasoning 刷新频率 |
| **OpenViking** (ByteDance) | 分层知识树，分级检索 (L0→L1→L2) | Session commit 时自动提取 6 类记忆 |
| **Mem0** | 服务端 LLM 事实提取，语义搜索 + reranking | 自动去重，服务端管理 |
| **Hindsight** | 知识图谱，实体解析，多策略检索 | `hindsight_reflect` 做跨记忆合成 |
| **Holographic** | 本地 SQLite + FTS5，Trust Scoring，HRR 代数查询 | 非对称反馈 (+0.05/-0.10)，矛盾自动检测 |
| **RetainDB** | 混合搜索 (Vector+BM25+Reranking)，7 种记忆类型 | Delta 压缩，importance 分级 |

### 7.3 DeerFlow 2.0 记忆系统详解

**项目**: bytedance/deer-flow — "Deep Exploration and Efficient Research Flow"

#### 核心架构：三层上下文 + Fact 体系 + LLM 批量更新 + Debounce 队列

| 组件 | 实现 | 关键细节 |
|---|---|---|
| **User Context** | 三段式 | workContext / personalContext / topOfMind，各有 summary + updatedAt |
| **History Context** | 三段式 | recentMonths / earlierContext / longTermBackground |
| **Facts** | 结构化条目 | id / content / category / confidence(0-1) / source / sourceError |
| **MemoryMiddleware** | after_agent hook | 每次 Agent 执行完自动触发记忆更新流程 |
| **MemoryUpdateQueue** | Debounce 队列 | 可配置 debounce_seconds，同 thread 多次更新覆盖合并 |
| **MemoryUpdater** | LLM 结构化更新 | 输出 shouldUpdate + newFacts[] + factsToRemove[] 的 JSON diff |
| **存储** | JSON 文件 | `.deer-flow/memory.json` |
| **API** | RESTful | GET/POST/PATCH/DELETE /api/memory/facts，完整 CRUD |
| **配置** | 可调参数 | enabled / max_facts / confidence_threshold / max_injection_tokens / debounce_seconds |

#### DeerFlow 关键设计决策

1. **Confidence 分数** — 每个 Fact 有 0-1 的置信度，低于阈值不注入。比"要么有要么没有"更细粒度。
2. **sourceError 字段** — Fact 可以记录"之前错误的做法"，Agent 能学到"不要做什么"。
3. **纠正/强化检测** — 用正则匹配用户话语中的纠正信号（"that's wrong"、"你理解错了"、"重试"）和正向强化（"perfect"、"对，就是这样"），检测到后在 prompt 中加强引导。
4. **Debounce 机制** — 记忆更新有 debounce_seconds 防止高频写入，同 thread 覆盖合并。
5. **上传文件过滤** — 自动剥离 `<uploaded_files>` 块，从记忆中清除文件上传相关句子（因为上传文件是 session 级临时的）。
6. **max_facts 硬淘汰** — 超出上限时按 confidence 排序截断，低置信度直接丢弃。
7. **完整 API 层** — 有 import/export/clear/reload，适合前端管理界面。

#### DeerFlow LLM 更新 Prompt 核心逻辑（源码）

```
Before extracting facts, perform a structured reflection:
1. Error/Retry Detection → category "correction", confidence >= 0.95
2. User Correction Detection → 记录正确做法 + sourceError 字段
3. Project Constraint Discovery → 记录为适当 category

Fact Categories:
- preference: 工具、风格、方法偏好
- knowledge: 专业知识、掌握的技术
- context: 背景事实（职位、项目、位置）
- behavior: 工作模式、沟通习惯
- goal: 目标、学习计划
- correction: 明确的错误纠正（confidence >= 0.95）

输出: JSON { user: {shouldUpdate...}, history: {shouldUpdate...},
             newFacts: [...], factsToRemove: [...] }
```

## 八、记忆更新机制对比（源码级分析）

### 8.1 OpenClaw — Agent 自驱动 + Memory Flush Hook

**更新触发三层机制**：

1. **Agent 主动写入** — 在对话中 Agent 自己 `edit/write` MEMORY.md、memory/日志文件
2. **Memory Flush** — 核心机制，在 session 上下文接近爆满时自动触发：
   - `softThresholdTokens`（默认 4000 tokens）— 当 session token 数超过 `contextWindow - softThreshold`，触发 memory flush
   - `forceFlushTranscriptBytes` — 当 transcript 原始字节超过阈值时强制触发
   - 每个 compaction 周期只跑一次（`memoryFlushCompactionCount` 追踪，防止重复 flush）
   - Flush 时用独立的 LLM 调用（有专用 system prompt），把当前对话提炼成记忆写入文件
3. **Heartbeat 维护** — AGENTS.md 指引 Agent 在心跳期间定期回顾日志，提炼 MEMORY.md

**更新质量控制**：
- ❌ 无结构化约束，全靠 Agent prompt 指导
- ❌ 无安全扫描（记忆内容不过滤）
- ✅ Flush 前的 LLM 调用有独立 prompt 引导提炼方向
- ✅ 向量混合检索（v2）支持 MMR 去重

### 8.2 Hermes Agent — 有界容量 + 被迫合并 + 安全扫描

**更新触发**：

1. **Agent 主动调用 memory tool** — `add`/`replace`/`remove` 三个 action，子串匹配定位
2. **Proactive Learning** — Agent prompt 指导"发现偏好、纠正、环境信息时主动保存"
3. **外部 Provider 自动提取** — 如 Mem0 每轮后自动 LLM 抽取 facts；Honcho 每轮后同步对话

**更新质量控制**：
- ✅ **硬容量上限** — 满了报错，Agent 被迫 consolidate
- ✅ **安全扫描** — prompt injection / credential exfiltration / invisible Unicode 检测
- ✅ **去重** — 完全重复内容自动拒绝
- ✅ **子串匹配替换** — replace/remove 只需给唯一子串

**冻结快照时序**：
```
Session Start → 加载 MEMORY.md → 注入 system prompt → 整个 session 不变
                                                            ↓
                  Agent 写入 memory tool → 立即写盘 → 下个 session 才生效
```
- 好处：保护 LLM prefix KV cache，不用每个 turn 重新计算 system prompt
- 代价：当前 session 看不到刚存的记忆（tool 返回值里能看到 live state）

### 8.3 DeerFlow — LLM 批量更新 + Debounce 队列 + 信号检测

**更新触发三层机制**：

1. **MemoryMiddleware（`after_agent` hook）**：
   - 过滤消息：只保留 user input + final assistant response（去掉 tool calls）
   - 检测纠正信号：正则匹配 `"that's wrong"`、`"你理解错了"`、`"重试"` 等
   - 检测正向强化：`"perfect"`、`"exactly right"`、`"对，就是这样"` 等
   - 纠正/强化 flag 入队 MemoryUpdateQueue

2. **MemoryUpdateQueue — Debounce 队列（源码）**：
   ```python
   # 同 thread 多次更新 → 覆盖合并（保留最新消息，合并 correction/reinforcement flag）
   self._queue = [c for c in self._queue if c.thread_id != thread_id]
   self._queue.append(context)
   
   # Debounce timer 重置
   self._timer = threading.Timer(config.debounce_seconds, self._process_queue)
   
   # 队列间隔 0.5s 处理，防 rate limit
   if len(contexts_to_process) > 1:
       time.sleep(0.5)
   ```

3. **MemoryUpdater — LLM 生成结构化 JSON diff（源码）**：
   ```python
   # 把当前 memory JSON + 对话文本 → LLM → 输出更新指令
   prompt = MEMORY_UPDATE_PROMPT.format(
       current_memory=json.dumps(current_memory, indent=2),
       conversation=conversation_text,
       correction_hint=correction_hint,  # 纠正检测时注入额外引导
   )
   
   # LLM 输出 JSON:
   # { user: {workContext: {shouldUpdate, summary}, ...},
   #   history: {recentMonths: {shouldUpdate, summary}, ...},
   #   newFacts: [{content, category, confidence, sourceError}],
   #   factsToRemove: ["fact_id_1"] }
   ```

| 维度 | OpenClaw | Hermes Agent | DeerFlow |
|---|---|---|---|
| **更新触发** | Agent 自写 + Memory Flush hook | Agent memory tool + 外部 Provider | Middleware 自动 + Debounce 队列 |

## 九、记忆衰减策略对比（源码级分析）

### 9.1 OpenClaw — Temporal Decay + Hybrid Search（最接近学术标准）

从 `memory-search.d.ts` 源码看到 OpenClaw v2 有完整的向量检索 + 衰减系统：

```typescript
// OpenClaw 记忆搜索配置（源码）
query: {
  maxResults: number,
  minScore: number,               // 最低分数线过滤
  hybrid: {
    enabled: boolean,
    vectorWeight: number,          // 向量搜索权重
    textWeight: number,            // 文本搜索权重
    candidateMultiplier: number,   // 候选集倍率
    mmr: {                         // Maximal Marginal Relevance 去重
      enabled: boolean,
      lambda: number,              // 相关性 vs 多样性权衡
    },
    temporalDecay: {
      enabled: boolean,            // 时间衰减开关
      halfLifeDays: number,        // 半衰期（天数）
    },
  },
}
```

**衰减公式**: `relevance_score = base_score × 0.5^(age_days / halfLifeDays)`

**特点**:
- 半衰期模型 — 旧记忆自然淡化，新记忆权重高
- 混合检索 — 向量相似度 + FTS 文本匹配加权
- MMR 去重 — 防止搜索结果中记忆条目过于相似
- 最低分数线 — `minScore` 过滤低质量结果

### 9.2 Hermes Agent — 容量压力淘汰 + Provider 各异

Hermes **不做自动衰减**，策略是利用有限空间施加进化压力：

**内置记忆**：
```
容量满了 → 报错 → Agent 被迫选择"什么值得保留"
                  ↓
          ┌── 方案 A: 删除过时条目 (remove)
          ├── 方案 B: 合并相关条目 (replace)
          └── 方案 C: 压缩冗余表述 (replace with shorter version)
```
- 本质是让 LLM 做价值判断：没有时间衰减公式，完全靠 Agent judgment
- 优点：简单、Agent 有自主权、不会误删重要记忆
- 缺点：Agent 可能做出糟糕的合并决策

**外部 Provider 衰减策略**：

| Provider | 衰减策略 |
|---|---|
| **Holographic** | Trust Scoring — 非对称反馈 helpful(+0.05) / unhelpful(-0.10)；矛盾检测 `contradict` action |
| **Honcho** | 辩证推理刷新 — `dialecticCadence` 控制 LLM reasoning 频率，自动更新/淡化过时用户模型 |
| **Hindsight** | 多策略检索 — `recall_budget: low/mid/high` 控制搜索深度，`hindsight_reflect` 跨记忆合成 |
| **OpenViking** | 分级检索 — L0 (~100 tokens) → L1 (~2k) → L2 (full)，按需加载 |
| **RetainDB** | importance 分级 + delta 压缩 |

### 9.3 DeerFlow — Confidence 淘汰 + max_facts 硬上限

**源码中的淘汰逻辑**：

```python
# DeerFlow updater.py — _apply_updates() 源码
# 超出 max_facts 时：按 confidence 排序，保留 top N
if len(current_memory["facts"]) > config.max_facts:
    current_memory["facts"] = sorted(
        current_memory["facts"],
        key=lambda f: f.get("confidence", 0),
        reverse=True,
    )[: config.max_facts]
```

**注入时的排序逻辑**：

```python
# DeerFlow prompt.py — format_memory_for_injection() 源码
# 按 confidence 降序排列，逐条计算 token 直到超出 budget
ranked_facts = sorted(facts, key=lambda f: f.get("confidence"), reverse=True)
for fact in ranked_facts:
    line_tokens = _count_tokens(line_text)  # tiktoken 精确计数
    if running_tokens + line_tokens <= max_tokens:
        fact_lines.append(line)
    else:
        break  # 超出 budget 就截断
```

**⚠️ 没有时间衰减**（DeerFlow 团队已在文档中确认）：
> MEMORY_IMPROVEMENTS.md: "TF-IDF similarity-based fact retrieval — **Planned / not yet merged**"
> "Previous versions of this document described TF-IDF/context-aware retrieval as if it were already shipped. **That was not accurate.**"

DeerFlow 目前只有 confidence-based 淘汰，**没有基于时间或使用频率的衰减**。一个半年前 confidence=0.9 的 fact 依然会排在新的 0.8 fact 前面。

## 十、对 AI 中台记忆设计的改进建议

> 基于三方对比，对第四章设计的补充和修正。

### 10.1 从 Hermes 借鉴：有界记忆 + 安全扫描

**改进 Memory Writer（§4.1）**：

1. **入库安全扫描** — 参考 Hermes 的安全扫描机制，在 `_deduplicate()` 之前增加：
   ```python
   async def _security_scan(self, entry: MemoryEntry) -> bool:
       """写入前安全检查"""
       patterns = [
           r'ignore\s+previous\s+instructions',    # prompt injection
           r'curl.*\$.*KEY',                        # credential exfiltration
           r'cat\s+\.env',                          # secret file access
       ]
       for pattern in patterns:
           if re.search(pattern, entry.content, re.IGNORECASE):
               return False
       # 不可见 Unicode 检测
       if re.search(r'[\u200b-\u200f\u2028-\u202f\ufeff]', entry.content):
           return False
       return True
   ```

2. **每用户记忆容量上限** — 参考 Hermes 有界记忆思路，给 memories 表加 per-user 上限：
   ```python
   # 满时自动触发提炼合并，而非直接拒绝
   MAX_ACTIVE_MEMORIES_PER_USER = 500
   if active_count > MAX_ACTIVE_MEMORIES_PER_USER:
       await self.distiller.force_distillation(tenant_id, owner_id)
   ```

### 10.2 从 DeerFlow 借鉴：纠正检测 + Debounce + sourceError

   ```python
   _CORRECTION_PATTERNS = [
       r'\bthat(?:\'s| is) (?:wrong|incorrect)\b',

       r'\bperfect(?:[.!?]|$)',
   ```

2. **Debounce 队列** — 参考 DeerFlow 的 `MemoryUpdateQueue`，LLM 提取不要每轮都做：

   ```python
   # 在 Worker 层加 debounce，同 session 多轮对话批量处理
   ```

   ```sql
   ALTER TABLE memories ADD COLUMN correction_note TEXT;  -- "之前错误地用了 rm 而不是 trash"
   ```

### 10.3 从 OpenClaw 借鉴：时间衰减 + 冻结快照

**改进 Memory Retriever（§4.2）**：

1. **强化时间衰减** — 当前设计的半衰期 7 天太激进。参考 OpenClaw 的 `halfLifeDays` 可配置思路：
   ```python
   # 不同 tier 使用不同半衰期
   HALF_LIFE_DAYS = {
       "short_term": 7,     # 短期记忆 7 天半衰期
       "long_term": 90,     # 长期记忆 90 天半衰期
       "permanent": None,   # 永久记忆不衰减
   }
   ```

2. **冻结快照优化** — 参考 Hermes 的 frozen snapshot，对高频 Agent（如客服 Agent）：
   ```python
   # 每个 session 开始时做一次记忆快照，session 内不刷新
   # 好处：保护 LLM prefix KV cache，减少检索调用
   # 适用：短对话场景（客服、问答）
   # 不适用：长对话场景（研究、编程），需要实时记忆
   session_memory_strategy: "frozen" | "live"
   ```

**改进 Memory Distiller（§4.3）**：

3. **综合衰减评分** — 结合三方优点的衰减公式：
   ```python
   def compute_memory_score(memory: MemoryEntry) -> float:
       """
       综合评分 = confidence × temporal_decay × access_boost
       
       - confidence: 初始置信度（DeerFlow 思路）
       - temporal_decay: 时间衰减（OpenClaw 思路）
       - access_boost: 使用频率加成（Holographic trust 思路）
       """
       # 时间衰减（可配置半衰期）
       age_days = (now - memory.created_at).days
       half_life = HALF_LIFE_DAYS.get(memory.memory_tier, 30)
       if half_life is None:
           temporal_decay = 1.0
       else:
           temporal_decay = 0.5 ** (age_days / half_life)
       
       # 使用频率加成（被检索越多越重要）
       access_boost = min(1.0 + math.log1p(memory.access_count) * 0.1, 2.0)
       
       # 综合评分
       return memory.importance * temporal_decay * access_boost
   ```

## 十一、睡眠记忆巩固 (Sleep Consolidation)

### 11.1 设计动机

```
神经科学背景:
  人类睡眠期间，海马体会对白天的情节记忆进行"重播"（Hippocampal Replay），
  将重要的短期记忆巩固为长期记忆，同时修剪无用连接。

  具体机制:
  ① 慢波睡眠 (SWS)  — 情节记忆 → 语义记忆（提炼抽象模式）
  ② REM 睡眠        — 跨记忆关联（发现意想不到的联系）
  ③ 突触稳态假说    — 白天突触增强，睡眠期间全局缩放恢复基线（剪枝）

AI 中台类比:
  ① 低峰期（凌晨）深度运行，不影响在线请求
  ② 三阶段流水线: 重播强化 → 模式提炼 → 剪枝归档
  ③ 跨 Session / 跨 Agent 整合 — Distiller 只做单用户单轮，Sleep 做全局

与 §4.3 MemoryDistiller 的区别:
┌─────────────────┬──────────────────────────────┬──────────────────────────────┐
│                 │  MemoryDistiller (§4.3)       │  SleepConsolidator (本章)    │
├─────────────────┼──────────────────────────────┼──────────────────────────────┤
│ 触发            │ 定时 (每 6 小时)               │ 低峰期检测 (凌晨业务低谷)     │
│ 粒度            │ 单用户短期记忆聚类             │ 全租户跨 Session/Agent        │
│ 目标            │ 短期 → 长期 tier 提升          │ 情节 → 语义/程序记忆升维      │
│ 关联            │ 无                            │ 构建 memory_links 关系图      │
│ 强化            │ 无                            │ 重播高价值记忆，更新 embedding │
│ 组织记忆        │ 无                            │ 挖掘租户级共性模式 → org 记忆 │
│ 成本            │ 低（快速聚类）                 │ 高（多 LLM 调用，低峰运行）   │
└─────────────────┴──────────────────────────────┴──────────────────────────────┘
```

### 11.3 低峰期调度器 (SleepConsolidator)

```python
# memory/sleep_consolidator.py

class SleepConsolidator:
    """
    睡眠记忆巩固调度器。

    调度策略:
    - 默认在租户时区凌晨 2:00-4:00 运行（可配置）
    - 基于活跃请求数做二次检测（避免业务高峰误触发）
    - 每个租户独立调度，互不影响
    - 单次运行有时间上限（max_duration_minutes），超时优雅中断
    """

    def __init__(self, config: SleepConfig):
        self.config = config
        self._running: bool = False

    async def maybe_run(self, tenant_id: str) -> Optional[ConsolidationReport]:
        """检查是否满足运行条件，满足则执行"""

        # ── 防重入 ──
        if self._running:
            return None

        # ── 低峰期检测 ──
        if not await self._is_off_peak(tenant_id):
            return None

        # ── 冷却期检测（距上次运行不足 N 小时不再跑）──
        last_run = await self.store.get_last_consolidation(tenant_id)
        if last_run:
            elapsed_hours = (datetime.utcnow() - last_run.finished_at).seconds / 3600
            if elapsed_hours < self.config.min_interval_hours:
                return None

        return await self._run(tenant_id)

    async def _is_off_peak(self, tenant_id: str) -> bool:
        """
        低峰期判断（两个条件都满足才触发）:
        1. 当前时间在租户时区的配置窗口内（默认 02:00-04:00）
        2. 近 5 分钟的请求 QPS < off_peak_qps_threshold
        """
        tenant_cfg = await self.store.get_tenant_config(tenant_id)
        tz = pytz.timezone(tenant_cfg.timezone or "Asia/Shanghai")
        local_now = datetime.now(tz)

        in_window = (
            self.config.off_peak_start_hour
            <= local_now.hour
            < self.config.off_peak_end_hour
        )
        if not in_window:
            return False

        recent_qps = await self.metrics.get_qps(tenant_id, window_minutes=5)
        return recent_qps < self.config.off_peak_qps_threshold

    async def _run(self, tenant_id: str) -> ConsolidationReport:
        self._running = True
        started_at = datetime.utcnow()
        report = ConsolidationReport(tenant_id=tenant_id, started_at=started_at)

        try:
            # 获取该租户下所有有活跃记忆的 owner (user/agent)
            owners = await self.store.list_active_memory_owners(tenant_id)

            deadline = started_at + timedelta(minutes=self.config.max_duration_minutes)

            for owner in owners:
                if datetime.utcnow() > deadline:
                    logger.warning("Sleep consolidation timeout for tenant=%s", tenant_id)
                    report.timed_out = True
                    break

                owner_report = await self._consolidate_owner(tenant_id, owner)
                report.owner_reports.append(owner_report)

            # 租户级：组织记忆挖掘
            if self.config.enable_org_pattern_mining:
                org_report = await self._mine_org_patterns(tenant_id)
                report.org_report = org_report

        finally:
            self._running = False
            report.finished_at = datetime.utcnow()
            await self.store.save_consolidation_log(report)

        return report

    async def _consolidate_owner(self, tenant_id: str,
                                  owner: MemoryOwner) -> OwnerReport:
        """对单个 owner 依次执行四个 Phase"""
        report = OwnerReport(owner=owner)

        # Phase 1: 重播强化
        report.replay = await self.replayer.replay(tenant_id, owner)

        # Phase 2: 模式提炼（情节 → 语义/程序）
        report.patterns = await self.pattern_extractor.extract(tenant_id, owner)

        # Phase 3: 记忆关联图更新
        report.links = await self.linker.update_links(tenant_id, owner)

        # Phase 4: 重要度重标定 + 剪枝
        report.recalib = await self.recalibrator.recalibrate(tenant_id, owner)

        return report
```

### 11.4 Phase 1：记忆重播 (MemoryReplayer)

```
人类类比: SWS 慢波睡眠期间海马体重播情节序列，将重要经历的突触连接反复激活。

AI 实现:
  - 识别"高价值但近期未被检索"的记忆（可能因为 embedding 漂移而命中率下降）
  - 用最新 embedding 模型重新计算向量（刷新 embedding）
  - 将这些记忆暂时提升检索权重（access_boost 加成）
  - 找到与当前对话主题高度相关但从未被关联的记忆（"潜在相关"）
```

```python
# memory/replayer.py

class MemoryReplayer:
    """
    记忆重播 — 刷新 embedding，提升高价值记忆的检索命中率。

    触发条件（OR 逻辑）:
    1. 记忆重要度 >= replay_importance_threshold，但近 N 天未被检索
    2. Embedding 模型版本落后（embedding_model != current_model）
    3. 记忆距上次 embedding 刷新超过 refresh_days
    """

    async def replay(self, tenant_id: str,
                     owner: MemoryOwner) -> ReplayReport:
        report = ReplayReport()

        # ── 选取需要重播的记忆 ──
        candidates = await self._select_replay_candidates(tenant_id, owner)
        report.total_candidates = len(candidates)

        for mem in candidates:
            try:
                # 刷新 embedding（使用最新模型）
                new_embedding = await self.embedder.embed(
                    mem.content,
                    model=self.config.current_embedding_model,
                )

                # 计算新旧 embedding 的余弦距离，衡量语义漂移
                drift = cosine_distance(mem.embedding, new_embedding)

                await self.store.update_embedding(
                    memory_id=mem.memory_id,
                    embedding=new_embedding,
                    embedding_model=self.config.current_embedding_model,
                    last_replayed_at=datetime.utcnow(),
                )

                report.refreshed += 1
                if drift > self.config.drift_alert_threshold:
                    report.high_drift_memories.append((mem.memory_id, drift))
                    logger.info(
                        "Memory drift detected: id=%s drift=%.3f", mem.memory_id, drift
                    )

            except Exception as e:
                logger.warning("Replay failed for memory_id=%s: %s", mem.memory_id, e)
                report.failed += 1

        return report

    async def _select_replay_candidates(self, tenant_id: str,
                                         owner: MemoryOwner) -> List[MemoryEntry]:
        """
        候选选取策略 — 三类候选取并集
        """
        # 类型 A: 高重要度 + 长时间未检索（高价值但被遗忘）
        type_a = await self.store.list_memories(
            tenant_id=tenant_id,
            owner_id=owner.owner_id,
            memory_tier=["long_term", "permanent"],
            status="active",
            min_importance=self.config.replay_importance_threshold,  # 默认 0.7
            not_accessed_days=self.config.replay_stale_days,         # 默认 14
            limit=50,
        )

        # 类型 B: Embedding 版本过旧
        type_b = await self.store.list_memories(
            tenant_id=tenant_id,
            owner_id=owner.owner_id,
            embedding_model_not=self.config.current_embedding_model,
            status="active",
            limit=100,
        )

        # 类型 C: 超过 refresh_days 未刷新 embedding
        type_c = await self.store.list_memories(
            tenant_id=tenant_id,
            owner_id=owner.owner_id,
            last_replayed_before=datetime.utcnow() - timedelta(
                days=self.config.embedding_refresh_days  # 默认 30
            ),
            status="active",
            limit=100,
        )

        # 去重合并
        seen = set()
        candidates = []
        for mem in [*type_a, *type_b, *type_c]:
            if mem.memory_id not in seen:
                seen.add(mem.memory_id)
                candidates.append(mem)

        return candidates
```

### 11.5 Phase 2：模式提炼 (PatternExtractor)

```
人类类比: 睡眠期间大脑将多次经历的情节记忆压缩为抽象的语义/程序记忆。
  - 多次遇到"猫"的情节 → 形成"猫"的语义概念
  - 多次成功执行某操作 → 内化为程序性技能（不需要有意识回忆步骤）

AI 实现:
  - 扫描 N 天内同一 owner 的情节记忆 (episodic)
  - 聚类相关情节（基于向量相似度）
  - LLM 从情节簇中提炼模式 → 新的语义(semantic)/程序(procedural)记忆
  - 将原始情节标记为 distilled，保留引用链
```

```python
# memory/pattern_extractor.py

class PatternExtractor:
    """
    模式提炼 — 从情节记忆中提炼语义/程序记忆。

    与 MemoryDistiller._distill_cluster() 的区别:
    - Distiller: 同类记忆合并去重（多条相似 factual → 一条 factual）
    - PatternExtractor: 情节升维（N条 episodic → 1条 procedural/semantic）
    """

    async def extract(self, tenant_id: str,
                      owner: MemoryOwner) -> PatternReport:
        report = PatternReport()

        # ── 收集近期情节记忆 ──
        episodic_memories = await self.store.list_memories(
            tenant_id=tenant_id,
            owner_id=owner.owner_id,
            content_type="episodic",
            status="active",
            created_after=datetime.utcnow() - timedelta(
                days=self.config.pattern_window_days  # 默认 30 天
            ),
            min_importance=0.3,
            limit=200,
        )

        if len(episodic_memories) < self.config.min_cluster_size:
            return report  # 情节记忆太少，不做提炼

        # ── 向量聚类 ──
        clusters = self._cluster_embeddings(
            episodic_memories,
            min_cluster_size=self.config.min_cluster_size,  # 默认 3
            similarity_threshold=0.75,
        )

        report.total_clusters = len(clusters)

        for cluster in clusters:
            pattern = await self._extract_pattern(cluster, owner)
            if not pattern:
                continue

            # 检查是否已存在类似的 semantic/procedural 记忆
            existing = await self.store.search_similar(
                embedding=pattern.embedding,
                tenant_id=tenant_id,
                owner_id=owner.owner_id,
                content_types=["semantic", "procedural"],
                threshold=0.90,
                limit=1,
            )

            if existing and existing[0].score > 0.90:
                # 已有类似规律，强化它（提升 importance）
                boost = min(existing[0].item.importance * 1.15, 1.0)
                await self.store.update_importance(existing[0].item.memory_id, boost)
                report.strengthened += 1
            else:
                # 新规律，入库
                pattern.memory_tier = "long_term"
                pattern.source_type = "sleep_pattern_extraction"
                pattern.embedding = await self.embedder.embed(pattern.content)

                new_id = await self.store.insert(pattern)

                # 建立情节 → 规律的提炼引用
                for episodic_mem in cluster:
                    await self.store.insert_memory_link(
                        source_id=episodic_mem.memory_id,
                        target_id=new_id,
                        link_type="distilled_into",
                    )

                report.new_patterns += 1

        return report

    async def _extract_pattern(self, cluster: List[MemoryEntry],
                                owner: MemoryOwner) -> Optional[MemoryEntry]:
        """用 LLM 从情节簇中提炼规律"""

        episodes_text = "\n".join([
            f"- [{m.created_at.strftime('%Y-%m-%d')}] {m.content}"
            for m in sorted(cluster, key=lambda x: x.created_at)
        ])

        prompt = f"""以下是同一用户在不同时间发生的相关情节记忆，请提炼出其中蕴含的规律或经验：

{episodes_text}

提炼规则:
1. 识别这些情节背后的共同模式或行为规律
2. 如果是操作类规律 → type: "procedural"（例如："每次查询 CRM 需要先指定 region 参数"）
3. 如果是知识类规律 → type: "semantic"（例如："该用户负责华南区业务，客户主要在广东"）
4. 不要复述情节，要提炼抽象的规律
5. 如果这些情节没有明显规律，返回 null

返回 JSON（无规律时返回 null）:
{{"content": "...", "type": "procedural|semantic", "tags": [...], "importance": 0.0-1.0}}"""

        try:
            response = await self.llm.chat(
                model=self.config.pattern_model or "qwen-turbo",
                messages=[{"role": "user", "content": prompt}],
                temperature=0,
                max_tokens=200,
            )

            data = json.loads(response.content)
            if not data:
                return None

            return MemoryEntry(
                content=data["content"],
                content_type=data["type"],
                tags=data.get("tags", []),
                importance=data.get("importance", 0.7),
                owner_type=owner.owner_type,
                owner_id=owner.owner_id,
            )
        except Exception as e:
            logger.warning("Pattern extraction failed: %s", e)
            return None

    def _cluster_embeddings(self, memories: List[MemoryEntry],
                             min_cluster_size: int,
                             similarity_threshold: float) -> List[List[MemoryEntry]]:
        """
        基于向量相似度的简单聚类（Single-linkage）。
        足够快，不需要 sklearn，内存友好。
        """
        clusters: List[List[MemoryEntry]] = []
        assigned = set()

        for i, mem_i in enumerate(memories):
            if i in assigned:
                continue

            cluster = [mem_i]
            assigned.add(i)

            for j, mem_j in enumerate(memories):
                if j in assigned or j == i:
                    continue
                if mem_i.embedding is None or mem_j.embedding is None:
                    continue

                sim = cosine_similarity(mem_i.embedding, mem_j.embedding)
                if sim >= similarity_threshold:
                    cluster.append(mem_j)
                    assigned.add(j)

            if len(cluster) >= min_cluster_size:
                clusters.append(cluster)

        return clusters
```

### 11.6 Phase 3：记忆关联图 (MemoryLinker)

```
人类类比: REM 睡眠时大脑在看似不相关的记忆间建立联系（创意顿悟来源）。

AI 实现:
  - 找出同一 owner 下语义相近但从未被关联的记忆对
  - 建立 memory_links 表，记录记忆间的关系类型
  - 检索时可沿关联链做 Graph RAG 式扩散检索
  - 跨 Agent 关联：用户与 Agent-A 建立的经验，对 Agent-B 也有参考价值
```

```python
# memory/linker.py

class MemoryLinker:
    """
    记忆关联图构建 — 发现并记录记忆间的隐性关联。

    关联类型:
    - related_to     : 语义相关（余弦相似度 0.70-0.85，弱关联）
    - reinforces     : 相互强化（两条记忆互相支持，余弦 > 0.85）
    - contradicts    : 矛盾（语义相近但内容对立，需 LLM 判断）
    - distilled_into : 情节 → 规律（Phase 2 建立）
    - superseded_by  : 被取代（由 Writer 建立，此处不新增）
    - cross_agent    : 跨 Agent 关联（用户在不同 Agent 建立的关联记忆）
    """

    async def update_links(self, tenant_id: str,
                           owner: MemoryOwner) -> LinkReport:
        report = LinkReport()

        # ── 获取该 owner 所有活跃的长期/永久记忆 ──
        memories = await self.store.list_memories(
            tenant_id=tenant_id,
            owner_id=owner.owner_id,
            memory_tier=["long_term", "permanent"],
            status="active",
            limit=500,
        )

        if len(memories) < 2:
            return report

        # ── 找出尚未关联的记忆对中，语义相近的候选 ──
        new_links = await self._discover_links(memories, tenant_id)
        report.discovered = len(new_links)

        # ── 批量写入 memory_links ──
        for link in new_links:
            await self.store.upsert_memory_link(link)

        report.inserted = len(new_links)

        # ── 跨 Agent 关联（同用户不同 Agent 的记忆） ──
        if owner.owner_type == "user":
            cross_agent_links = await self._discover_cross_agent_links(
                tenant_id, owner.owner_id
            )
            for link in cross_agent_links:
                await self.store.upsert_memory_link(link)
            report.cross_agent_links = len(cross_agent_links)

        return report

    async def _discover_links(self, memories: List[MemoryEntry],
                               tenant_id: str) -> List[MemoryLink]:
        """批量向量比对，发现相似但未关联的记忆对"""
        links = []
        existing_pairs = await self.store.get_existing_link_pairs(
            [m.memory_id for m in memories]
        )

        # 分批做向量内积（避免 O(n²) 全量比对）
        BATCH = 50
        for i in range(0, len(memories), BATCH):
            batch_i = memories[i:i + BATCH]
            for j in range(i + 1, len(memories), BATCH):
                batch_j = memories[j:j + BATCH]
                for mi in batch_i:
                    for mj in batch_j:
                        pair_key = tuple(sorted([mi.memory_id, mj.memory_id]))
                        if pair_key in existing_pairs:
                            continue
                        if mi.embedding is None or mj.embedding is None:
                            continue

                        sim = cosine_similarity(mi.embedding, mj.embedding)

                        if sim >= 0.85:
                            # 强关联或潜在矛盾，需 LLM 判断
                            link_type = await self._classify_link(mi, mj)
                            links.append(MemoryLink(
                                source_id=mi.memory_id,
                                target_id=mj.memory_id,
                                link_type=link_type,
                                similarity=sim,
                            ))
                        elif sim >= 0.70:
                            links.append(MemoryLink(
                                source_id=mi.memory_id,
                                target_id=mj.memory_id,
                                link_type="related_to",
                                similarity=sim,
                            ))

        return links

    async def _classify_link(self, mi: MemoryEntry,
                              mj: MemoryEntry) -> str:
        """对高相似度记忆对，用 LLM 判断是强化还是矛盾"""
        prompt = f"""判断以下两条记忆的关系:

记忆A: {mi.content}
记忆B: {mj.content}

只返回以下之一:
- "reinforces"  （两条互相支持、一致）
- "contradicts" （两条内容矛盾或冲突）
- "related_to"  （相关但无明确支持/矛盾关系）"""

        try:
            response = await self.llm.chat(
                model="qwen-turbo",
                messages=[{"role": "user", "content": prompt}],
                temperature=0,
                max_tokens=20,
            )
            result = response.content.strip().strip('"')
            if result in ("reinforces", "contradicts", "related_to"):
                return result
        except Exception:
            pass
        return "related_to"
```

### 11.7 Phase 4：重要度重标定 (ImportanceRecalibrator)

```python
# memory/recalibrator.py

class ImportanceRecalibrator:
    """
    重要度重标定 — 综合使用频率、时间衰减、关联度重新计算 importance。

    §10.3 中定义的综合评分公式在此处批量执行，并对低分记忆执行归档。
    """

    async def recalibrate(self, tenant_id: str,
                          owner: MemoryOwner) -> RecalibReport:
        report = RecalibReport()

        memories = await self.store.list_memories(
            tenant_id=tenant_id,
            owner_id=owner.owner_id,
            status="active",
            memory_tier=["short_term", "long_term"],  # permanent 不重标定
            limit=1000,
        )

        now = datetime.utcnow()
        updates: List[Tuple[int, float]] = []

        for mem in memories:
            new_score = self._compute_score(mem, now)

            if abs(new_score - mem.importance) < 0.01:
                continue  # 变化极小，跳过

            if new_score < self.config.archive_threshold:  # 默认 0.05
                # 分数过低，归档
                await self.store.archive(mem.memory_id)
                report.archived += 1
            else:
                updates.append((mem.memory_id, new_score))

        # 批量更新
        if updates:
            await self.store.bulk_update_importance(updates)
            report.updated = len(updates)

        # 强制提炼（容量超限时）
        active_count = await self.store.count_active_memories(tenant_id, owner.owner_id)
        if active_count > self.config.max_active_memories_per_owner:
            await self.distiller.force_distillation(tenant_id, owner.owner_id)
            report.forced_distillation = True

        return report

    def _compute_score(self, mem: MemoryEntry, now: datetime) -> float:
        """§10.3 综合衰减评分: confidence × temporal_decay × access_boost"""

        # 时间衰减（可配置半衰期，permanent 不衰减）
        half_life_map = {
            "short_term": self.config.short_term_half_life_days,   # 默认 7
            "long_term":  self.config.long_term_half_life_days,    # 默认 90
        }
        half_life = half_life_map.get(mem.memory_tier)
        if half_life:
            age_days = max((now - mem.created_at).days, 0)
            temporal_decay = 0.5 ** (age_days / half_life)
        else:
            temporal_decay = 1.0

        # 使用频率加成（被检索越多越重要，上限 2.0）
        access_boost = min(1.0 + math.log1p(mem.access_count) * 0.1, 2.0)

        # 关联数加成（有更多关联的记忆更重要）
        link_count = mem.link_count or 0
        link_boost = min(1.0 + math.log1p(link_count) * 0.05, 1.3)

        return min(mem.importance * temporal_decay * access_boost * link_boost, 1.0)
```

### 11.8 组织级模式挖掘 (OrgPatternMiner)

```python
# memory/org_pattern_miner.py

class OrgPatternMiner:
    """
    租户级共性模式挖掘 — 从多个用户的记忆中发现组织级知识。

    触发场景:
    - 多个用户都踩过同一个坑 → 写入 org 记忆，所有用户自动受益
    - 多个用户使用同一工具都有相同技巧 → 提炼为 procedural org 记忆

    隐私保护:
    - 只挖掘 procedural / semantic 记忆（不含个人信息的 factual/episodic 不参与）
    - 挖掘前对内容做 PII 检测，发现 PII 直接跳过该条记忆
    - 生成的 org 记忆不含任何用户 ID
    """

    async def mine(self, tenant_id: str) -> OrgMiningReport:
        report = OrgMiningReport(tenant_id=tenant_id)

        # ── 收集全租户的 procedural / semantic 长期记忆 ──
        # 只取最近 30 天内活跃的，避免挖掘过时信息
        memories = await self.store.list_memories_across_users(
            tenant_id=tenant_id,
            content_types=["procedural", "semantic"],
            memory_tier=["long_term", "permanent"],
            status="active",
            created_after=datetime.utcnow() - timedelta(
                days=self.config.org_mining_window_days  # 默认 30
            ),
            min_importance=0.6,
            limit=500,
        )

        # ── PII 过滤 ──
        memories = [m for m in memories if not m.pii_detected]

        if len(memories) < self.config.org_min_samples:  # 默认至少 5 条
            return report

        # ── 聚类（同类 procedural 簇） ──
        clusters = self._cluster_by_topic(memories, similarity_threshold=0.80)

        for cluster in clusters:
            # 至少 N 个不同用户都有该类记忆，才提炼为 org 记忆
            unique_users = len(set(m.owner_id for m in cluster))
            if unique_users < self.config.org_min_users:  # 默认 3 个用户
                continue

            # 检查是否已存在类似 org 记忆
            representative = cluster[0]
            existing = await self.store.search_similar(
                embedding=representative.embedding,
                tenant_id=tenant_id,
                owner_type="org",
                threshold=0.88,
                limit=1,
            )

            if existing:
                # 强化已有 org 记忆（提升 importance，更新 access 统计）
                boost = min(existing[0].item.importance * 1.1, 1.0)
                await self.store.update_importance(existing[0].item.memory_id, boost)
                report.strengthened_org_memories += 1
            else:
                # 生成新的 org 记忆（去除用户信息）
                org_memory = await self._synthesize_org_memory(cluster, unique_users)
                if org_memory:
                    org_memory.owner_type = "org"
                    org_memory.owner_id = f"org:{tenant_id}"
                    org_memory.memory_tier = "permanent"
                    org_memory.data_classification = "internal"
                    org_memory.embedding = await self.embedder.embed(org_memory.content)

                    await self.store.insert(org_memory)
                    report.new_org_memories += 1

        return report
```
