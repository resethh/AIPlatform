## 1. MCP 原子工具 vs Skill 实现 Schedule Event

**时间：** 2026-04-25 08:03

**Q：** Schedule event 这个工具，是用 MCP 包装几个原子工具实现，还是用 Skill 实现比较好？

**A：** 两种路径各有适用场景：

### MCP 原子工具组合

适合：**底层能力封装，给多个 Agent/Skill 复用**

- 把日历 API 拆成独立工具：`create_event`、`update_event`、`delete_event`、`list_events`、`get_freebusy`
- 每个工具职责单一，参数清晰，LLM 自己决定调用顺序
- 优点：**组合灵活**，其他 Skill 也能调用这些工具
- 缺点：多步操作时 LLM 可能犯错（漏参数、顺序搞反），token 消耗多

### Skill 封装

适合：**固定流程、需要可靠执行的复合操作**

- 在 SKILL.md 里写清楚流程指引，调用底层 MCP 工具或直接走脚本
- 比如"创建日程"这个 Skill 可以规定：先查冲突 → 确认参与人 → 创建 → 设提醒，一套固定步骤
- 优点：**流程可控**，减少 LLM 瞎发挥的空间
- 缺点：灵活性低一点，新需求要改 Skill

### 建议：两层都做

```
┌─────────────────────────┐
│   Skill: schedule-event  │  ← 流程编排 + 错误处理 + 用户交互指引
│   (SKILL.md + scripts)   │
├─────────────────────────┤
│   MCP Tools (原子层)      │  ← create_event / list_events / freebusy...
│   (独立 MCP server)       │
└─────────────────────────┘
```

**简单判断标准：**
- 单步、无歧义 → 裸 MCP 工具就够
- 多步流程、需要判断逻辑 → 包一层 Skill
- 两种场景都有 → 两层都做，Skill 调 MCP

## 2. MCP 能否实现复合工具（内部编排多个原子操作）

**时间：** 2026-04-25 08:08

**Q：** 是不是可以在 MCP 里也写上调多个工具的一个图？

**A：** **完全可以，而且很多场景下这是更好的做法。**

### MCP 里做复合工具

```
MCP Server
├── create_event          ← 原子工具
├── list_events           ← 原子工具
├── get_freebusy          ← 原子工具
└── schedule_smart_event  ← 复合工具（内部串联多个步骤）
```

复合工具内部逻辑示例：

```python
def schedule_smart_event(title, attendees, preferred_time, duration):
    # 1. 查参与人忙闲
    busy = get_freebusy(attendees, preferred_time)
    # 2. 找到共同空闲时段
    slot = find_available_slot(busy, preferred_time, duration)
    # 3. 创建日程
    event = create_event(title, slot, attendees)
    # 4. 设置提醒
    add_reminder(event.id, minutes_before=15)
    return event
```

### 优劣分析

**✅ 优点：**
- **确定性高** — 代码逻辑，不靠 LLM 推理，不会漏步骤
- **省 token** — 一次调用搞定，不用来回多轮
- **速度快** — 服务端内部调用，没有多次 LLM ↔ MCP 往返
- **易测试** — 纯代码，写单测就行

**⚠️ 代价：**
- **灵活性下降** — 流程固定在代码里，改需求要改代码重部署
- **参数设计更复杂** — 要提前想好各种边界情况
- **LLM 可见性低** — 中间步骤对 LLM 不透明

### 实际建议：分层暴露

```
MCP Server
├── create_event              ← 原子（灵活组合时用）
├── list_events               ← 原子
├── get_freebusy              ← 原子
└── schedule_smart_event      ← 复合（常用场景一键搞定）
```

- **复合工具** 覆盖 80% 的常见场景
- **原子工具** 保留，给 LLM 处理边角情况的灵活度

本质是 API 设计里经典的"低级 API + 高级 API"模式。

## 3. Skill、MCP 和 Agent 各自适合什么场景

**时间：** 2026-04-25 08:41

**Q：** Skill、MCP 和 Agent 都适合什么场景？

**A：**

### 层级关系

```
┌─────────────────────────────────────────────┐
│              Agent (智能体)                    │
│  有目标、能决策、能自主行动                       │
│  ┌─────────────────────────────────────┐     │
│  │           Skill (技能)                │     │
│  │  固定流程指引，约束 Agent 行为          │     │
│  │  ┌─────────────────────────────┐     │     │
│  │  │      MCP Tools (工具)        │     │     │
│  │  │  原子能力，被调用的"手和脚"    │     │     │
│  │  └─────────────────────────────┘     │     │
│  └─────────────────────────────────────┘     │
└─────────────────────────────────────────────┘
```

### 🔧 MCP Tool — "手和脚"

- **本质：** 一个可调用的函数/API，输入 → 输出，无决策能力
- **适合：** 单步确定性操作、被多个 Agent/Skill 复用的底层能力
- **例子：** `create_event()`, `send_message()`, `query_database()`
- **特点：** 没有主动性，被调用才干活

### 📋 Skill — "操作手册"

- **本质：** 一份给 Agent 看的 SOP，用自然语言 + 约束规则写成
- **适合：** 多步流程 + 需要 LLM 判断和用户交互、约束 Agent 行为边界、固化领域知识
- **例子：** `wecom-schedule`, `email-assistant`, `coding-agent`
- **特点：** 本身不执行操作，是给 Agent 的"指南"

### 🤖 Agent — "有脑子的人"

- **本质：** 有目标、能感知环境、能自主决策和行动的实体
- **适合：** 目标模糊需要拆解规划、多轮推理、跟人或其他 Agent 协作
- **例子：** 综合助手、PM Agent、编码 Agent
- **特点：** 有自主性，能规划，能犯错也能纠错

### 对比表

| 维度 | MCP Tool | Skill | Agent |
|------|----------|-------|-------|
| 比喻 | 锤子、扳手 | 操作手册 | 工人 |
| 决策能力 | ❌ 没有 | ❌ 没有（指导决策） | ✅ 有 |
| 确定性 | 高（代码逻辑） | 中（LLM 遵循度） | 低（自主推理） |
| 灵活性 | 低 | 中 | 高 |
| 调用关系 | 被 Agent/Skill 调用 | 被 Agent 加载参考 | 自己驱动一切 |
| 改动成本 | 改代码重部署 | 改 Markdown 即时生效 | 改 prompt/配置 |
| Token 消耗 | 最低 | 中 | 最高 |

### 选型决策树

```
你的任务是什么？
├── 单步、输入输出明确？ → MCP Tool
├── 多步但流程固定、不需要 LLM 判断？ → MCP 复合工具（代码编排）
├── 多步、需要 LLM 判断/用户交互/追问？ → Skill（调 MCP 工具）
├── 目标模糊、需要规划拆解？ → Agent（加载 Skill，调 MCP 工具）
└── 需要多角色协作？ → 多 Agent（各自有 Skill + 工具）
```

**总结：MCP 是能力，Skill 是规矩，Agent 是脑子。三层配合，缺一不可。**

## 4. AI 中台 Agent 构建：单例 vs 工厂模式

**时间：** 2026-04-25 12:51

**Q：** 我想使用 LangChain 方法创建 Agent，在 AI 中台项目中，我应该用单例还是工厂模式？

**A：** **工厂模式为主，单例做缓存层。**

AI 中台核心是多租户、多场景服务，不同用户/业务线需要不同配置的 Agent。LangChain Agent 天然有状态（memory、chat history、tool binding），多用户共享实例会串上下文。

### 推荐架构：工厂 + 缓存池

```python
class AgentFactory:
    def __init__(self, tool_registry, llm_registry):
        self.tool_registry = tool_registry
        self.llm_registry = llm_registry
        self._template_cache = {}

    def create_agent(self, config: AgentConfig) -> AgentExecutor:
        llm = self.llm_registry.get(config.model_name)
        tools = [self.tool_registry.get(t) for t in config.tool_names]
        memory = self._create_memory(config)
        agent = create_react_agent(llm=llm, tools=tools, prompt=config.prompt)
        return AgentExecutor(agent=agent, tools=tools, memory=memory)
```

### 各组件复用策略

| 组件 | 模式 | 原因 |
|------|------|------|
| LLM Client | 单例/连接池 | 底层 HTTP 连接复用，无用户状态 |
| Tool 实例 | 单例（无状态工具） | 搜索、计算等通用工具共用 |
| Agent 模板 | 单例缓存 | prompt + tool 组合不变 |
| AgentExecutor | 工厂（每次新建） | 有 memory/state，必须隔离 |
| Memory | 工厂（每会话新建） | 聊天历史用户级别隔离 |
| AgentFactory | 单例 | 全局一个工厂就够 |

### 会话管理

多轮对话场景，AgentExecutor 需要会话池管理：

```python
class SessionManager:
    def __init__(self, factory: AgentFactory, ttl: int = 1800):
        self.factory = factory
        self.sessions = {}  # session_id → AgentExecutor
        self.ttl = ttl      # 30分钟过期

    def get_or_create(self, session_id, config):
        if session_id in self.sessions:
            return self.sessions[session_id]
        executor = self.factory.create_agent(config)
        self.sessions[session_id] = executor
        return executor
```

## 5. Agent 构造参数深度分析

**时间：** 2026-04-25 12:54

**Q：** 从 Agent 所需的构造参数分析：系统提示词、记忆、工具、Skill

**A：**

### 5.1 系统提示词 (System Prompt)

- **状态性：** 无状态，纯文本
- **复用策略：** 单例缓存（模板固定，变量部分每次渲染）
- **改动方式：** 改了 prompt 刷新缓存即可

```python
class PromptRegistry:
    _cache = {}

    @classmethod
    def get(cls, template_id: str, **variables) -> str:
        if template_id not in cls._cache:
            cls._cache[template_id] = load_from_db(template_id)
        return cls._cache[template_id].format(**variables)
```

### 5.2 记忆 (Memory)

- **状态性：** 强状态，每条消息都在改变
- **隔离要求：** 用户绑定 + 会话绑定，**绝不共享**
- **复用策略：** 工厂模式，每会话新建

两层记忆设计：

```
┌─────────────────────────────────┐
│  长期记忆 (Long-term Memory)      │  ← 用户级，跨会话
│  用户偏好、历史摘要、关键事实       │
│  存储：Redis / DB                 │
├─────────────────────────────────┤
│  短期记忆 (Short-term Memory)     │  ← 会话级，单次对话
│  当前对话的 chat_history           │
│  存储：内存 / Redis with TTL      │
└─────────────────────────────────┘
```

### 5.3 工具 (Tools)

分两类处理：
- **无状态工具**（搜索、计算、天气）→ 单例，全局共享
- **有状态工具**（带用户凭证的 API）→ 工厂，注入用户 context

```python
class ToolRegistry:
    def __init__(self):
        self._stateless = {}
        self._factories = {}

    def get_tools(self, tool_names, user_ctx):
        tools = []
        for name in tool_names:
            if name in self._stateless:
                tools.append(self._stateless[name])
            elif name in self._factories:
                tools.append(self._factories[name](user_ctx))
        return tools
```

### 5.4 Skill

- **状态性：** 无状态，是静态"配方"
- **本质：** Prompt 片段 + Tool 清单 + 流程约束
- **复用策略：** 纯单例缓存，加载一次到处复用

### 总览

```
Agent 构造 = Prompt + Memory + Tools + Skills
               │        │        │        │
               │        │        │        └── 单例缓存（静态配方）
               │        │        └── 无状态→单例 / 有状态→工厂
               │        └── 工厂（每会话新建，绝不共享）
               └── 单例缓存（模板固定，变量渲染）
```

**状态性排序：**
```
Memory ≫ Tools(有状态) > Prompt(变量) > Skill = Tools(无状态)
最不能共享 ──────────────────────────────────── 最能共享
```

**核心结论：Memory 是唯一必须隔离的，其他都能不同程度复用。设计中台时，把 Memory 的隔离做好，其他问题都不大。**
