### 如何理解 MCP 采用 Manager-Worker 分离架构
**问题**: 如何理解 MCP 采用 Manager-Worker 分离架构
**回答**: 核心是把 Agent 系统拆成两个不同信任域——Worker（污染区，LLM 驱动、易被注入）只拿可吊销的 consumer token；Manager + Gateway（洁净区）持有真实凭证。传统方案（加密存盘 / 环境变量 / 短期令牌）都无法解决"凭证进入 Agent 进程即泄露"的根本问题，只有物理分离 + 强可信代理才能切断链路。适用于多租户 ToB、生产环境高价值 API、合规审计场景；额外收获是细粒度吊销——Worker 被攻破只需吊销 token，无需轮换真实凭证
**相关页面**: [[应用网关设计方案]]

### Swagger/OpenAPI 导入遇到动态凭证怎么办
**问题**: 从 Swagger/OpenAPI 导入，如果是要动态获取的 key 呢
**回答**: Gateway 角色从"代理静态凭证"升级为"凭证获取与生命周期管理器"，Worker 依然无感。通过 credential_provider 插件体系统一抽象 8 种典型认证（OAuth2 cc/ac、AWS STS/SigV4、JWT、Session、mTLS 等），Swagger 导入时解析 securitySchemes 自动选 provider。关键区分根凭证（Vault 长期）与运行时凭证（内存短期），缓存按服务级 / 用户级分粒度。已补入 mcp-management.md
**相关页面**: [[应用网关设计方案]]

### Skill、MCP、Agent 三者的区别
**问题**: skill、mcp、agent 之间的区别是什么
**回答**: 三者不在同一层——**Agent** = 执行主体（谁做）/ **MCP** = 工具接入协议（能做什么）/ **Skill** = 场景化程序性知识（怎么做好）。

核心差异：

| 维度 | Agent | MCP | Skill |
|---|---|---|---|
| **本质** | 跑 LLM + 上下文 + 工具循环的**执行进程** | 把外部系统能力标准化为工具的**通信协议** | 教 Agent 如何组合工具完成一类任务的**知识包** |
| **粒度** | 一个会话 / 一个任务 | 一个 API 端点 = 一个工具（原子动作） | 一个场景 = 一个 Skill（工作流） |
| **回答的问题** | "谁在执行？" | "有什么工具可用？" | "这类任务应该怎么做？" |
| **典型形态** | Hermes Worker / DeerFlow Lead / 中台 AgentWorker | `create_issue_from_thread` / `get_customer` 这类工具定义 | `SKILL.md` + 引用的工具列表 + 经验说明 |
| **演进速度** | 框架级，改动慢 | 接口级，稳定 | 场景级，可频繁迭代 |
| **治理重心** | 运行时资源 / 安全边界 | 权限 + 凭证 + 速率限制 | 审核发布 + 版本灰度 + 条件激活 |

协作关系：

```
Agent（执行主体）
  ├─ 用 MCP 工具（原子能力）：调 CRM、查数据库、发消息
  └─ 按 Skill 指引（程序性知识）：这类客服场景先查 CRM，再查工单，再回复
```

Anthropic 原话锚点：**MCP 给 Agent 提供工具，Skill 教 Agent 如何用这些工具完成真实工作**。最厉害的 Agent 两者都用——MCP 负责能力边界，Skill 负责场景专家化。

各自适合的问题：
- 要接入一个新外部系统 → 建 MCP Server（[[应用网关设计方案]]）
- 要让 Agent 在某类任务上更专业 → 写 Skill（[[skills/skill-manage.md]]）
- 要让任务能持续自主运行 → 部署 Agent（[[architecture/hermes.md]] / [[architecture/deerflow.md]]）

边界规则（本知识库已确立的分工原则，见 [[应用网关设计方案]] §"MCP 与 Skill 的定位差异"）：
- **权限治理**归 MCP 层——工具级 RBAC、凭证隔离、速率限制
- **场景演进**归 Skill 层——自由迭代、审核发布、沙箱评估
- **执行调度**归 Agent 框架——上下文管理、记忆、工具循环

混淆导致的麻烦：
- 把权限逻辑写进 Skill → 审计链断了
- 把场景逻辑写进 MCP 工具 → 接口稳定性垮了
- 把执行细节写进 MCP 或 Skill → 框架级升级时全面改造

**相关页面**: [[应用网关设计方案]]、[[skills/skill-manage.md]]、[[architecture/hermes.md]]

## 历史技术参考（来源：raw/hermes-QA.md）

> 以下条目来自 Hermes 源码探索记录，非用户提问，仅作技术参考。

### Hermes MCP server 工具注册和调用链路
Skill 工具与 MCP 工具共用 `ToolRegistry` 单例；MCP server 长驻（asyncio Task，指数退避重试）；调用链：`registry.dispatch()` → `_run_on_mcp_loop()` → `run_coroutine_threadsafe()`，主线程 0.1s 轮询支持中断。

### Hermes Skill 评估、加载、降级机制
`skill_view()` 做十步评估返回 `SkillReadinessStatus`；`build_skills_system_prompt()` 走三层缓存后三重过滤注入提示；`fallback_for_tools` 是纯 Python 过滤，LLM 看不到被隐藏的 Skill；三层缓存内容不同（LRU 存 prompt 字符串，磁盘快照存原始元数据，L3 直接读文件）。

### Hermes 凭证管理机制
静态 API key 存 `~/.hermes/.env`（原子写入，`chmod 0o600`）；OAuth token 存 `~/.hermes/auth.json`，调用前检查是否即将过期（提前 120s 刷新）；凭证文件注册用 `ContextVar` 防多会话串漏，拒绝绝对路径 + 路径穿越。

### Hermes Agent 完整生命周期与主循环设计
四阶段：工具注册（进程内一次）→ tools schema 固定（每实例一次）→ 主循环 `run_conversation()`（最多 90 次 API 调用）→ `finish_reason=stop` 退出；MCP 工具格式 `mcp_{server}_{tool}`，跨线程桥接。
