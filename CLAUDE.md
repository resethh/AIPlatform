
## 项目概览

这是一个专注于 **AI 系统架构与 Agent 工程** 的个人知识库。
关注领域：系统架构设计、Agent 安全、记忆管理、Skill 迭代机制、上下文工程。
此仓库的 `wiki/` 完全由 Claude 负责编写和维护，用户只提供原始资料和问题。

## 关键的三层架构

### 第一层: `raw/` (原始资料 - 只读)
- **目的**: 存放不同方向的技术报告。
- **规则**: **严禁修改或删除** `raw/` 中的任何内容。这是证据来源。

### 第二层: `wiki/` (编译产物 - 完全由 AI 维护)
- **目的**: 编译后的架构知识图谱。Claude 负责编写和维护。
- **结构**: 纯 Markdown 文件。使用 `[[双向链接]]` 关联组件、模式与漏洞。
- **每个模块目录下，页面分两类**:
    - **开源框架分析页**（命名如 `hermes-xxx.md`、`deerflow-xxx.md`、`openclaw-xxx.md`）：记录 Hermes / OpenClaw / DeerFlow 各自的技术方案，原汁原味，不掺杂中台设计。
    - **中台设计页**（命名如 `xxx.md`，无框架前缀）：基于开源框架的参考与对比，输出我们自己的 AI 中台设计方案，明确写出"参考了哪个框架的哪个设计"。
    - 两类页面通过 `[[双向链接]]` 关联，中台设计页应注明参考来源。

### 第三层: `CLAUDE.md` (规则文件 - 本文件)
- **目的**: 约定 Claude 的行为规范、工作流程和 wiki 写作标准。
- **规则**: 用户可修改，Claude 严格遵守，不得自行篡改。

## 具体工作流指令

### 1. 收录新资料
当用户说"帮我收录"、"处理一下"、"把这个加进去"等类似意思时，执行以下步骤：
- **找到新文件**: 运行 `git status --short raw/` 和 `git diff --name-only --diff-filter=A HEAD raw/`，找出 `raw/` 下**未跟踪 (`??`) 或新增 (`A`) 的文件**。没有新文件就告诉用户，不要继续。
- **读懂内容**: 仔细阅读，提炼出涉及的**架构模式**、**安全边界**、**记忆机制**或**上下文策略**等核心要点。
- **找准归属位置**:
    1. 先看 `wiki/` 现有目录，判断内容往哪个模块放（`architecture/`、`security/`、`memory/`、`context/` 等）。
    2. 没有合适的模块，就在 `wiki/` 下新建子目录，同时在 `wiki/INDEX.md` 里登记新模块。
    3. **判断文件类型**：
       - 内容是 Hermes / OpenClaw / DeerFlow 的源码分析或技术报告 → 写入或更新**开源框架分析页**（如 `hermes-memory.md`）。
       - 内容是我们自己的设计方案（署名"大迈"，标题含"AI 中台"、"企业级"等）→ 写入或更新**中台设计页**（如 `memory-management.md`），并在文中注明参考了哪个框架的哪个设计。
- **逐一更新**:
    1. 更新 `wiki/architecture/` 里的架构模式页。
    2. 更新 `wiki/security/` 里的安全攻击面或防御措施页。
    3. 更新 `wiki/memory/` 或 `wiki/context/` 的相关页面。
    4. 更新 `wiki/INDEX.md` 和 `wiki/log.md`。

### 2. 文档补齐
当用户说"文档不够全面"、"列出功能清单"、"先列清单再补齐"、"哪些地方没写到"等类似意思时，执行以下步骤：
- **生成功能清单**: 对照对应的 `raw/` 原始文件，逐项列出该主题**应该覆盖的所有功能/机制**（用表格或列表，标注已覆盖 ✅ / 缺失 ❌）。
- **确认后补齐**: 把清单给用户看，确认范围后逐项补写缺失内容，更新对应 wiki 页面。
- **框架对比文档**: 如果补齐过程中涉及多个框架的横向比较，同步更新或新建 `wiki/comparison.md`，并登记到 `wiki/INDEX.md`。

### 3. 框架对比
当用户说"对比一下"、"不同框架的区别"、"横向比较"等类似意思时：
- 对比维度至少覆盖：架构风格、记忆系统、上下文管理、Skills/Tools、路由、安全、自我改进能力、部署模型。
- 结果写入或更新 `wiki/comparison.md`，登记到 `wiki/INDEX.md`。
- 每个对比项明确标注来源页面，如 `[[hermes-skills.md]]`。

### 4. 知识库巡检
当用户说"检查一下"、"巡检"、"有没有问题"等类似意思时，逐项排查：
- **前后矛盾**: `wiki/` 各页面之间有没有说法打架的地方（比如 A 页说某组件是同步的，B 页说它是异步的）。
- **过期信息**: 有没有安全漏洞的判断已经过时，或者结论需要更新。
- **断链问题**: Skill 迭代记录里有没有引用了已废弃接口却没标注 `[deprecated]` 的地方。

### 5. 提问与解答
当用户提问时，按以下步骤处理：
- 先翻 `wiki/INDEX.md` 和 `wiki/architecture/decision-log.md`，找相关背景。
- **记录到 QA 文档，避免重复解答**:
    1. 打开 `wiki/QA.md`（没有就新建）。
    2. 看看已有的问题里有没有跟这次**意思差不多**的（主题相同或高度重叠）。
    3. 有类似问题，就把这次的回答**合并进去**，补充新角度或更新说法。
    4. 没有类似问题，按以下格式**追加一条**：
       ```
       ## Q: <问题摘要> (<YYYY-MM-DD>)
       **问题**: <完整问题>
       **答案**: <回答要点>
       **相关页面**: [[wiki页面链接]]
       ```
- 回答完之后，主动问用户要不要把这次的结论单独存成 `wiki/insights/YYYY-MM-DD-design-review.md`。

## Wiki 页面类型规范

> 每个模块下都有两类页面，命名规则：
> - **开源框架分析页**: `{框架名}-{主题}.md`，如 `hermes-memory.md`
> - **中台设计页**: `{主题}.md`，如 `memory-management.md`

### `wiki/architecture/`

**开源框架分析页**（`hermes-agent.md` / `deerflow.md` / `openclaw.md` 等）
```markdown
# {框架名} {组件/架构} 分析

> 来源: [[raw/xxx]] | 分析日期: YYYY-MM-DD

## 一句话定位
## 核心设计
## 关键实现细节
## 亮点与局限
## 对中台设计的启示
- 参考点: [[architecture/xxx.md]]
```

**中台设计页**（`gateway.md` / `agent-router.md` 等）
```markdown
# {组件名}

## 一句话定位
## 核心设计决策 (ADR)
- **决策** / **背景** / **后果**
## 参考来源
- 参考了 [[hermes-xxx.md]] 的 {具体设计}
- 参考了 [[deerflow-xxx.md]] 的 {具体设计}
## 上下游依赖
- 上游: [[组件A]]  下游: [[组件B]]
## 关键接口/契约
```

## CLAUDE.md 自我优化规则

**每次回答用户问题后**，判断这次对话是否暴露了 CLAUDE.md 的规则空白。满足以下任一条件，就在本次回复末尾说明，并**直接修改 CLAUDE.md**：

| 触发信号 | 说明 | 对应动作 |
|---|---|---|
| 用户纠正了我的做法 | 说明有行为没有被规则约束 | 把正确做法写成新规则 |
| 用户要求了一个没有对应流程的操作 | 如"列功能清单"、"做对比" | 新增对应工作流指令 |
| 用户问题隐含了命名规范、模板或分类方式 | 如"框架分析页"和"中台设计页"的区分 | 补充到对应的页面类型规范 |
| 执行过程中发现现有规则互相矛盾或遗漏 | 如收录流程忘了更新 log.md | 修正对应规则 |

**不需要修改 CLAUDE.md 的情况**：用户在问具体知识点、做 wiki 内容更新、日常提问——这些不属于行为规范层面的变化。

**修改时的原则**：
- 只修改工作流指令和规范章节，不改项目概览和三层架构描述。
- 新规则用口语化表达（同现有风格），不写技术黑话。
- 修改完后告知用户"已更新 CLAUDE.md：新增了 XX 规则"。

## 回复风格要求

- **语言**: 全程中文，专有名词保留英文（如 Agent、ADR、Skill）。
- **长度**: 能用一句话说清楚的不写两句；操作步骤用编号列表。
- **态度**: 直接给结论，不做不必要的铺垫或总结性废话。
- **不确定时**: 明确说"我不确定"，不要编造。
