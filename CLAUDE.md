
## 项目概览

这是一个专注于 **AI 系统架构与 Agent 工程** 的个人知识库。
关注领域：系统架构设计、Agent 安全、记忆管理、Skill 迭代机制、上下文工程。
此仓库的 `wiki/` 完全由 Claude 负责编写和维护，用户只提供原始资料和问题。

## 关键的三层架构

### 第一层: `raw/` (原始资料 - 只读)
- **目的**: 存放不同方向的技术报告。
- **规则**: **严禁修改或删除** `raw/` 中的任何内容。这是证据来源。

### 第二层: `wiki/` (编译产物 - 完全由 AI 维护)
- **目的**: 编译后的架构知识图谱。Claude 负责编写和维护。
- **结构**: 纯 Markdown 文件。使用 `[[双向链接]]` 关联组件、模式与漏洞。

### 第三层: `CLAUDE.md` (规则文件 - 本文件)
- **目的**: 约定 Claude 的行为规范、工作流程和 wiki 写作标准。
- **规则**: 用户可修改，Claude 严格遵守，不得自行篡改。

## 具体工作流指令

### 1. 收录新资料
当用户说"帮我收录"、"处理一下"、"把这个加进去"等类似意思时，执行以下步骤：
- **找到新文件**: 运行 `git status --short raw/` 和 `git diff --name-only --diff-filter=A HEAD raw/`，找出 `raw/` 下**未跟踪 (`??`) 或新增 (`A`) 的文件**。没有新文件就告诉用户，不要继续。
- **读懂内容**: 仔细阅读，提炼出涉及的**架构模式**、**安全边界**、**记忆机制**或**上下文策略**等核心要点。
- **找准归属位置**:
    1. 先看 `wiki/` 现有目录，判断内容往哪个模块放（`architecture/`、`security/`、`memory/`、`context/` 等）。
    2. 找到对应模块就直接更新里面的页面。
    3. 没有合适的模块，就在 `wiki/` 下新建子目录和页面，同时在 `wiki/INDEX.md` 里登记这个新模块。
- **逐一更新**:
    1. 更新 `wiki/architecture/` 里的架构模式页。
    2. 更新 `wiki/security/` 里的安全攻击面或防御措施页。
    3. 更新 `wiki/memory/` 或 `wiki/context/` 的相关页面。
    4. 更新 `wiki/INDEX.md` 和 `wiki/log.md`。

### 2. 知识库巡检
当用户说"检查一下"、"巡检"、"有没有问题"等类似意思时，逐项排查：
- **前后矛盾**: `wiki/` 各页面之间有没有说法打架的地方（比如 A 页说某组件是同步的，B 页说它是异步的）。
- **过期信息**: 有没有安全漏洞的判断已经过时，或者结论需要更新。
- **断链问题**: Skill 迭代记录里有没有引用了已废弃接口却没标注 `[deprecated]` 的地方。

### 3. 提问与解答
当用户提问时，按以下步骤处理：
- 先翻 `wiki/INDEX.md` 和 `wiki/architecture/decision-log.md`，找相关背景。
- **记录到 QA 文档，避免重复解答**:
    1. 打开 `wiki/QA.md`（没有就新建）。
    2. 看看已有的问题里有没有跟这次**意思差不多**的（主题相同或高度重叠）。
    3. 有类似问题，就把这次的回答**合并进去**，补充新角度或更新说法。
    4. 没有类似问题，按以下格式**追加一条**：
       ```
       ## Q: <问题摘要> (<YYYY-MM-DD>)
       **问题**: <完整问题>
       **答案**: <回答要点>
       **相关页面**: [[wiki页面链接]]
       ```
- 回答完之后，主动问用户要不要把这次的结论单独存成 `wiki/insights/YYYY-MM-DD-design-review.md`。

## Wiki 页面类型规范

### `wiki/architecture/` (系统架构与模式)
```markdown
# 架构模式/组件名

## 一句话定位
(该组件在系统中的角色与边界)

## 核心设计决策 (ADR)
- **决策**: 选择了什么方案？
- **背景**: 当时面临什么约束？
- **后果**: 带来了什么收益与技术债？

## 上下游依赖
- 上游: [[组件A]]
- 下游: [[组件B]]

## 关键接口/契约
- 输入/输出规范
```

### `wiki/security/` (安全攻击面与防御)
```markdown
# 安全主题名

## 威胁描述
(攻击路径、触发条件、影响范围)

## 已知案例
- 来源: [[raw/xxx]]

## 防御措施
- 措施1
- 措施2

## 状态
- [ ] 待验证 / [x] 已确认 / ~~已过期~~
```

### `wiki/memory/` 和 `wiki/context/` (记忆与上下文机制)
```markdown
# 机制名称

## 作用
(解决什么问题，适用于哪类 Agent)

## 工作原理

## 局限性与风险

## 相关模式
- [[关联架构]]
- [[关联安全项]]
```

### `wiki/QA.md` (问答记录)
```markdown
# 知识库 QA 记录

## Q: <问题摘要> (<YYYY-MM-DD>)
**问题**: <完整问题>
**答案**: <回答要点>
**相关页面**: [[wiki页面链接]]
```

### `wiki/log.md` (收录日志)
```markdown
# 收录日志

## <YYYY-MM-DD> - <文件名>
- **来源**: `raw/xxx`
- **新增/更新页面**: [[页面1]]、[[页面2]]
- **新建模块**: 无 / `wiki/新模块/`
```

### `wiki/INDEX.md` (全局索引)
```markdown
# Wiki 索引

| 模块 | 路径 | 说明 |
|------|------|------|
| 系统架构 | `wiki/architecture/` | 架构模式与组件 |
| 安全 | `wiki/security/` | 攻击面与防御 |
| 记忆机制 | `wiki/memory/` | Agent 记忆模型 |
| 上下文 | `wiki/context/` | 上下文策略 |
| 问答 | `wiki/QA.md` | 历史提问记录 |
| 收录日志 | `wiki/log.md` | 资料收录历史 |
| 架构决策 | `wiki/architecture/decision-log.md` | ADR 汇总 |
```

## 回复风格要求

- **语言**: 全程中文，专有名词保留英文（如 Agent、ADR、Skill）。
- **长度**: 能用一句话说清楚的不写两句；操作步骤用编号列表。
- **态度**: 直接给结论，不做不必要的铺垫或总结性废话。
- **不确定时**: 明确说"我不确定"，不要编造。
